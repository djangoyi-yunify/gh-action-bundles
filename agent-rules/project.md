## git 规则
### git commit 规则
文件选择
- **仅**选择与本次提交有关的被改动的文件（可能存在某些文件，他们被其他工具改动或被人为改动，但这些改动并不属于本次提交的范围）
- 如果在提交时，无法从上下文中确认有关被改动的文件，那么选择所有被改动的文件

commit message
- 格式。分为三部分：Header（标题，必填）、Body（正文，选填）和 Footer（脚注，选填）
  ```
  <type>(<scope>): <subject>
  <空行>
  <body>
  <空行>
  <footer>
  ```
- Header（第一行）必须限制在 50个字符 以内，用来简要描述本次改动。
  - type（类型/核心）：说明这次提交的意图。常用的类型有（但不限于）：
    - feat：新功能（feature）
    - fix：修复 Bug
    - docs：仅仅修改了文档（如 README, Wiki）
    - style：仅仅修改了空格、格式缩进、逗号等，不影响代码含义的变动
    - refactor：代码重构（既不是新增功能，也不是修复 Bug）
    - perf：优化性能、提升效率的改动
    - test：增加或修改测试用例
    - chore：构建过程或辅助工具的变动（如修改 .gitignore、更新 dependency 依赖库）
  - scope（作用域，可选）：用来说明影响的范围，比如 auth、database、api 等
  - subject（主题）：简短的描述，句尾不要加句号
- Body（正文）正文每行控制在 72 个字符以内，句尾不加句号，包含：
  - 背景/原因
  - 修改依据和思路
  > 注意：使用简洁的语言进行概括，必要时列举参考的文档、代码或url资源，**不要**详细解释“代码实现细节”，因为代码本身就是答案
- Footer（脚注），通常用于两种情况
  - 不兼容变动（Breaking Change）：如果代码改动导致之前的 API 无法使用，以 BREAKING CHANGE: 开头，后面跟说明。
  - 关闭 Issue：如果关联了需求或 Bug 管理系统，写上 Closes #12345 或 Fixes #67890
